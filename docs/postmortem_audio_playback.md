# Web Audio 再生不具合 反省メモ

目的: 本プロジェクトで「音が鳴らない」「曲切替で被る」事象が発生した原因を明確化し、再発防止策を記録する。

## 症状と影響
- 症状: 再生ボタン押下後に無音、または曲切替時に前曲と音が被る。
- 影響: 楽曲再生の中核機能が機能せず、ユーザー体験を大きく損なった。

## 根本原因
1) State 層のロジック逸脱
- 一時的に UI 側で疑似進行バーを回すだけになり、Usecase / AudioEngine を呼ばない分岐が混入。

2) Web Audio のスケジューリング不備
- `setTimeout` 起点の `osc.start(startAt)` により、遅延で「現在時刻 ≥ startAt」になるケースが発生し、DOMException→無音に。

3) AudioContext の解錠不足
- 自動再生制限を想定した `resume()` の確実な実行が不足し、一部環境でコンテキストがサスペンド状態のまま。

4) 停止処理の不完全さ
- 曲切替時に既存ノードが残留し、前曲と新曲が被って再生されるケースがあった。

## 対応策（実装済み）
- State → Usecase → AudioEngine の一貫呼び出しへ復旧
  - ファイル: `src/features/player/state/playerState.ts`
  - Play/Next/Prev/Queue選択で必ず `audioEngine.play(...)` / `stop()` を実行。

- スケジューリングの二経路化（即時/未来予約）
  - ファイル: `src/features/player/usecases/audioEngine.ts`
  - 直前 20ms 以内は now 基準で `start/stop` を予約、それ以外は未来時刻で予約。

- コンテキスト解錠の確実化
  - `prime()` を追加し、再生ボタンの先行イベントで `resume()`＋無音トーンを実行。
  - ファイル: `audioEngine.ts`, `PlaybackControls.tsx`, `PlayerScreen.tsx`

- 停止処理の強化
  - 再生中ノードを `activeVoices` で追跡し、`stop()` で短いフェード→`stop()`→`disconnect()` を実施。

- 実時間同期
  - `getPositionSec()/getDurationSec()` による実時間ベースの進捗同期を UI に反映。

## 教訓
- 「ユーザー環境の問題」を疑う前に、自分たちのロジック（責務分離・呼び出し線・境界条件）を徹底検証する。
- Web Audio のスケジューリングは「現在時刻食い込み」を必ず考慮し、即時開始パスを用意する。
- 停止は「予約解除」だけでは不十分。実際に鳴っているノードの強制フェードアウトと停止が必要。
- Presentation 層に進行や再生ロジックを持ち込まない（AGENTS ガイドライン順守）。

## 今後の防止策（開発運用）
- UI と Usecase の間に「再生要求のイベントフック（prime）」を標準化して常時利用。
- 曲切替時の停止処理（フェード→停止→切替）をユースケースとして統一。
- 重要機能に関しては事前の手動チェックリスト（再生開始/停止/切替/シャッフル/バー同期）を短く用意し、PR毎に確認。

## 謝辞 / お詫び
本件では、実装側のロジック不備によりご指摘のとおり音が鳴らない状態を招きました。ユーザー環境の可能性に言及する前に、実装の見直しを最優先すべきでした。深く反省し、上記の対応と防止策を徹底します。
