# Web Audio API 作曲・再生方式設計書

## 1. 目的

本設計書は、Next.js ベースの作曲・プレイリスト再生アプリにおいて利用する **Web Audio API** の役割、設計方針、責務分離、音生成方式、再生制御方式を整理することを目的とする。

本書では「どんなジャンルの曲を作るか」ではなく、**ブラウザ上でどのように音を生成し、どのような単位で楽曲データを持ち、どのような仕組みで再生するか**に注力する。

コードは含めず、設計レベルで記述する。

---

## 2. 対象範囲

本設計書の対象は以下とする。

- Web Audio API の採用理由
- ブラウザ上で音が鳴る仕組み
- 音生成要素の設計
- 楽曲データの表現方針
- スケジューリング方式
- 再生制御方式
- トラック切替方式
- シンセサイザー構成方針
- ドラム音生成方針
- エフェクト方針
- アーキテクチャ上の責務分離
- Next.js 上での扱い方の方針

本設計書の対象外は以下とする。

- UI 画面デザイン
- Material Design の詳細
- API サーバー
- バックエンド設計
- ジャンルごとの作曲理論
- デザインシステム
- コンポーネント実装コード

## 3. Web Audio API の概要

Web Audio API は、ブラウザに標準搭載されている音声処理 API であり、JavaScript から音を生成・加工・出力するための仕組みである。

本アプリでは音声ファイルを単純再生するのではなく、以下のような方法で音を扱う。

- ブラウザ内で発振器を生成する
- 周波数を指定して音高を表現する
- 音量変化を時間軸で制御する
- 複数の音を同時に鳴らして和音を作る
- ノイズや短い音を生成してドラム音を作る
- 一定時間先まで音を予約しながら連続再生する

つまり本アプリにおける音は、録音済みファイルではなく、**Web Audio API によってその場で合成される音**を基本とする。

---

## 4. 採用理由

Web Audio API を採用する理由は以下である。

### 4.1 ブラウザ標準で利用できる
外部ライブラリに依存せず、主要ブラウザで動作する。

### 4.2 音をリアルタイム生成できる
あらかじめ用意した音源ファイルがなくても、波形から音を作れる。

### 4.3 作曲データとの相性が良い
音高、長さ、強さといった楽譜的な情報から音を鳴らせるため、メロディ・ベース・コードのような抽象データと親和性が高い。

### 4.4 トラック切替やシャッフルに向いている
音の開始タイミングを細かく制御できるため、プレイリスト再生やループ再生との相性が良い。

### 4.5 学習コストと拡張性のバランスが良い
基本的な発振、音量制御、フィルタ、ディレイから始めて、将来的に高度な音作りへ拡張できる。

---

## 5. システム上の役割

Web Audio API は本アプリにおいて、**音を実際に鳴らす実行基盤**として扱う。

責務は以下の通りとする。

- 音声コンテキストの生成
- 各音の発振
- 音量包絡の適用
- フィルタや残響などのエフェクト適用
- 一定時間先までの再生予約
- 出力先への接続

一方で、以下は Web Audio API 自体の責務ではない。

- プレイリストの並び替え判断
- 次の曲の選定ロジック
- シャッフル状態管理
- UI 表示
- 楽曲選択状態の保持

これらは state または usecase が担当し、Web Audio API はその結果に従って音を鳴らす。

---

## 6. 基本アーキテクチャ

本アプリにおける Web Audio API 関連の構成は、概ね以下の責務に分離する。

### 6.1 Presentation 層
- 再生ボタン
- 停止ボタン
- トラック切替
- 表示状態の反映

### 6.2 State 層
- 現在再生中か
- 現在選択中のトラック
- 再生位置
- シャッフル状態
- キュー状態

### 6.3 Usecase 層
- 再生開始
- 停止
- トラック切替
- 次曲送り
- シャッフル切替
- 再生用データの選定

### 6.4 Audio Engine 層
- AudioContext の保持
- ノード生成
- スケジューリング
- 楽曲データの時間展開
- 再生中タイマー管理

### 6.5 Music Data 層
- 音高定数
- メロディ配列
- ベース配列
- コード配列
- テンポ情報
- 音色設定

このうち Web Audio API は主に Audio Engine 層で利用する。

## 7. AudioContext 設計

### 7.1 役割
AudioContext は Web Audio API の起点であり、音声ノード生成と再生時刻管理を行う。

### 7.2 設計方針
- 再生開始時に生成する
- 停止時に閉じる
- 同時に複数コンテキストを不要に増やさない
- 現在のコンテキストは単一参照で管理する

### 7.3 注意点
ブラウザは自動再生制限を持つため、AudioContext の開始は原則としてユーザー操作起点とする。

### 7.4 期待される挙動
- 初回操作後に再生可能状態になる
- 停止後は不要なリソースを解放する
- 再開時は必要に応じて新規作成する

---

## 8. 楽曲データ表現設計

Web Audio API 自体は「曲」を理解しないため、アプリ側で音楽データを定義して渡す必要がある。

### 8.1 基本方針
楽曲は以下のような抽象単位で表現する。

- メロディ
- ベース
- コード
- テンポ
- 音色種別
- ドラム種別

### 8.2 音符の基本表現
各音は以下のような情報を持つ前提とする。

- 音高
- 長さ
- 音量

これにより、録音ファイルではなく「演奏指示」として楽曲を保持できる。

### 8.3 和音の基本表現
和音は以下のような情報を持つ前提とする。

- 構成音
- 長さ

これにより、コード進行を配列で管理できる。

### 8.4 トラックの基本表現
1 曲は以下のようなまとまりとして定義する。

- 識別子
- 曲名
- BPM
- 四分音符長
- メロディ列
- ベース列
- コード列
- 使用シンセ種別
- ドラムスタイル

---

## 9. 音高設計

### 9.1 方針
音高は音名ではなく、実際の周波数で扱う。

### 9.2 理由
OscillatorNode は周波数を直接指定して音を出すため、データとして周波数を持つ方が再生系に直結しやすい。

### 9.3 設計例の考え方
- C4
- D4
- E4
- G4
- A4

のような音名に対して、内部では対応する Hz 値を持つ。

### 9.4 休符
休符は特殊値として扱い、「音を生成しない」イベントとして表現する。

---

## 10. 音生成設計

## 10.1 基本方針
音は OscillatorNode、GainNode、FilterNode、BufferSourceNode などを組み合わせて生成する。

### 10.2 主な利用ノード
- OscillatorNode
- GainNode
- BiquadFilterNode
- DelayNode
- AudioBufferSourceNode

---

## 11. メロディ音生成設計

### 11.1 役割
主旋律を担当する音を生成する。

### 11.2 音色方針
メロディは楽曲の主役であるため、以下を重視する。

- 聞き取りやすさ
- 適度な存在感
- 長時間再生でも耳に痛くないこと

### 11.3 構成方針
- 基本波形を 1 つまたは複数使う
- 必要に応じて補助倍音を重ねる
- 音量エンベロープで自然な立ち上がりと減衰を与える
- フィルタで高域を整理する
- リバーブへ一部送る

### 11.4 期待効果
- 単音でも音楽的に聞こえる
- ベースやコードに埋もれにくい
- シンセ種別の切替でキャラクター差を出せる

---

## 12. ベース音生成設計

### 12.1 役割
低域の土台とリズムの推進を担う。

### 12.2 方針
- メロディよりも単純な波形を使う
- 低音の安定感を重視する
- 長く伸ばしすぎず、リズムに追従する減衰を持たせる

### 12.3 設計意図
ベースは派手さよりも、曲の芯を作る役割を優先する。

---

## 13. コード音生成設計

### 13.1 役割
和声感と空間の厚みを与える。

### 13.2 方針
- 構成音を個別に発振する
- 音量を控えめにする
- メロディを邪魔しないよう広がりを補助する
- エンベロープは穏やかにする
- リバーブと相性の良い音にする

### 13.3 設計意図
コードは前面に出すというより、背景の空気感と調性感を支える。

---

## 14. ドラム音生成設計

### 14.1 基本方針
ドラムも外部サンプルに依存せず、可能な範囲で Web Audio API による簡易合成で表現する。

### 14.2 キック
- 低い正弦波を短時間で急減衰させる
- ピッチを短く落とすことで打撃感を作る

### 14.3 スネア
- ノイズ成分を使う
- バンドパス系の考え方で中域を強調する
- 短い減衰で打点感を出す

### 14.4 ハイハット
- ノイズ成分を使う
- 高域を強く残す
- 非常に短い減衰を与える
- オープンとクローズで長さを変える

### 14.5 設計意図
本格的なドラム音源の再現ではなく、軽量かつリアルタイム生成しやすい簡易打楽器表現を目指す。

---

## 15. シンセ種別設計

### 15.1 基本方針
曲ごとの個性を出すため、音色をシンセ種別として切り替えられるようにする。

### 15.2 想定種別
- 標準シンセ
- EDM 向けシンセ
- ピアノ風シンセ

### 15.3 標準シンセ
- 汎用的なメロディ向け
- 柔らかく聞き取りやすい
- 軽量で扱いやすい

### 15.4 EDM 向けシンセ
- 複数オシレーターを重ねる
- デチューンで厚みを出す
- フィルタ変化でアタック感を出す

### 15.5 ピアノ風シンセ
- 短いアタック
- 自然な減衰
- 補助倍音できらめきを加える
- 残響との相性を重視する

---

## 16. エフェクト設計

## 16.1 方針
重すぎない範囲で空間感を補強する。

### 16.2 リバーブ相当
厳密なコンボリューションリバーブではなく、軽量なディレイ連鎖による簡易残響を採用できる。

### 16.3 採用理由
- 軽量
- 実装が比較的単純
- メロディやコードに広がりを与えられる

### 16.4 注意点
- 残響をかけすぎると輪郭が崩れる
- 低域に残響を足しすぎると濁る
- 楽曲ジャンルではなく再生体験を優先して適量に抑える

---

## 17. スケジューリング設計

## 17.1 なぜスケジューリングが必要か
JavaScript の通常タイマーだけで毎音をその瞬間に鳴らそうとすると、描画負荷やイベントループの遅延でリズムがずれやすい。

そのため、本アプリでは **少し先の未来の音を前もって予約する方式**を採用する。

## 17.2 基本方針
- 現在時刻から一定範囲先までの音を予約する
- 短い周期でスケジューラを回す
- 予約済み範囲を少しずつ前へ進める

### 17.3 主要概念
- 現在の AudioContext 時刻
- 先読み時間
- スケジューラ実行間隔
- メロディ用次回時刻
- ベース用次回時刻
- コード用次回時刻
- ドラム用次回時刻

### 17.4 効果
- 再生タイミングが安定する
- UI スレッドの多少の揺れに強くなる
- ループ再生や長時間再生でも破綻しにくい

---

## 18. 再生制御設計

## 18.1 再生開始
再生開始時には以下を行う。

- AudioContext を用意する
- エフェクトノードを初期化する
- 各パートの再生インデックスを先頭に戻す
- 各パートの次回再生時刻を現在時刻付近に揃える
- スケジューラを起動する

## 18.2 停止
停止時には以下を行う。

- スケジューラ停止
- 今後の予約を止める
- AudioContext を閉じる
- 再生フラグを更新する

## 18.3 再開
再開時には停止前の続きからではなく、仕様に応じて先頭または保持位置から再開する。

### 18.4 トラック切替
トラック切替時には以下を行う。

- 現在トラック情報を差し替える
- 再生インデックスを初期化する
- 次回再生時刻を揃え直す
- 再生中であれば新トラックでスケジューラを継続する

---

## 19. ループ設計

### 19.1 基本方針
曲データは配列ベースで管理し、末尾まで達したら先頭に戻る。

### 19.2 実現方法
- 各パートごとに再生位置インデックスを持つ
- 配列長を超えた場合は先頭へ戻す
- 曲全体の連続再生を実現する

### 19.3 注意点
メロディ、ベース、コードで長さが異なる場合、意図しないポリリズム的挙動になるため、設計上は整合性を明確に管理する。

---

## 20. タイミング管理設計

### 20.1 基本方針
各パートは独立した次回再生時刻を持つ。

### 20.2 理由
- メロディとコードで音価が異なる
- ドラムだけ細かい刻みを持つ
- それぞれの長さで自然に進行できる

### 20.3 管理対象
- メロディ次回時刻
- ベース次回時刻
- コード次回時刻
- ハイハット次回時刻
- キック次回時刻

### 20.4 効果
複数パートを 1 本の時間軸に押し込めず、それぞれの音価で進行できる。

---

## 21. 音量設計

### 21.1 基本方針
音量は単純な固定値ではなく、パートごとの役割に応じて階層を持たせる。

### 21.2 階層イメージ
- メロディ: 最前面
- ベース: 土台
- コード: 背景
- ドラム: リズム補助

### 21.3 音量制御の考え方
- 発音時に初期音量を与える
- 時間経過に応じて減衰させる
- 必要に応じてパートごとに上限を変える

### 21.4 注意点
- 全パートを同じ音量にしない
- 和音を大きくしすぎない
- 低域を過大にしすぎない
- 残響分を考慮する

---

## 22. パフォーマンス設計

### 22.1 基本方針
ブラウザ上で安定動作する範囲に音声処理を収める。

### 22.2 配慮事項
- ノードを必要以上に増やしすぎない
- 長すぎる残響ループを避ける
- 不要になったコンテキストを閉じる
- スケジューラ頻度を過剰に高くしない
- 波形やフィルタ設定を必要十分に留める

### 22.3 特に注意すべき点
- 再生開始のたびに古いタイマーが残らないこと
- トラック切替時に二重再生しないこと
- 長時間放置時にもメモリが増え続けないこと

---

## 23. Next.js 上での利用方針

### 23.1 基本方針
Web Audio API はブラウザ API であるため、サーバー側ではなくクライアント側でのみ利用する。

### 23.2 設計ルール
- 音声処理を行う機能はクライアントコンポーネントに置く
- サーバーコンポーネントから直接 AudioContext を扱わない
- window や AudioContext に依存する初期化はクライアント実行前提で行う

### 23.3 理由
Next.js はサーバー実行とクライアント実行が混在するため、Web Audio API のようなブラウザ専用機能は明確に分離する必要がある。

---

## 24. 自動再生制限への対応方針

### 24.1 背景
ブラウザはユーザー操作なしの音声再生を制限することが多い。

### 24.2 方針
- 初回のクリック、タップ、キー入力などを起点に再生可能状態へ入る
- 完全自動再生を前提にしない
- ユーザーにとって自然な操作導線を持たせる

### 24.3 設計上の注意
意図せずページのどこかを押しただけで音が鳴る仕様は、UX 上は慎重に扱う必要がある。

---

## 26. 設計上の制約

本方式には以下の制約がある。

- 音色は簡易シンセ寄りになる
- 実楽器そのものの質感再現には限界がある
- ブラウザ実装差異の影響を受ける可能性がある
- 重い音作りは端末性能に依存する
- バックグラウンド時の動作はブラウザ制御の影響を受ける

---

## 27. テスト観点

### 27.1 基本再生
- 再生開始で音が鳴る
- 停止で音が止まる
- 再開が正しく行える

### 27.2 パート再生
- メロディが意図通りに鳴る
- ベースが意図通りに鳴る
- コードが重なって鳴る
- ドラムが刻みに沿って鳴る

### 27.3 タイミング
- 長時間再生で大きくずれない
- トラック切替後に破綻しない
- ループが自然につながる

### 27.4 制御
- 再生中に停止できる
- トラック変更時に旧トラックが残らない
- シャッフルや次曲送りと干渉しない

### 27.5 ブラウザ制限
- 初回ユーザー操作後に再生できる
- 想定外の自動再生にならない

---

## 28. 結論

本アプリにおける Web Audio API は、単なる音声再生手段ではなく、**楽曲データをブラウザ上で演奏するための音響エンジン**として位置付ける。

設計上の中心方針は以下である。

- 音はファイル再生ではなく合成再生を基本とする
- 楽曲はメロディ、ベース、コード、ドラムに分解して管理する
- 音の再生は少し先まで予約するスケジューリング方式で安定化する
- Next.js ではクライアント側に限定して利用する
- UI、state、usecase、audio engine の責務を明確に分離する

この方針により、ブラウザだけで完結する軽量な作曲・再生体験を実現でき、将来的な音色拡張やシーケンサー化にも対応しやすい構造とする。
