# Next.js 作曲・プレイリスト再生アプリ 設計書

## 1. 文書情報

- 文書名: 作曲・プレイリスト再生アプリ 設計書
- 対象フレームワーク: Next.js
- 想定利用環境: Web ブラウザ
- 作成方針:
  - API は作成しない
  - フロントエンド完結
  - Material Design 準拠
  - UI と state を分離する
  - ビジネスロジックは usecase に集約する

---

## 2. システム概要

本アプリは、ユーザーが楽曲を選択し、プレイリスト形式で再生できる Web アプリである。  
UI は Spotify や Apple Music のようなシンプルな再生画面を参考にしつつ、Material Design に準拠した見た目と操作性を採用する。

本アプリは以下を満たす。

- 楽曲の再生
- 次の曲へ送る操作
- シャッフル再生
- 次に再生される曲一覧を画面右側に表示
- 次曲リストの表示 / 非表示切替
- UI 層と state 管理層の分離
- ビジネスロジックを usecase に集約

なお、本アプリは API サーバーを持たず、データはフロントエンド内で管理する。

---

## 3. 目的

### 3.1 ユーザー目標
- 作成済みまたは用意済みの楽曲を快適に再生できる
- 直感的な操作で次曲送りやシャッフルができる
- 再生予定の曲順を一覧で確認できる
- 必要に応じて右側リストの表示を切り替えられる

### 3.2 開発目標
- 機能追加しやすい構成にする
- UI と状態管理、ビジネスロジックの責務を明確に分離する
- 将来的にループ再生、検索、複数プレイリスト管理などへ拡張しやすくする

## 6. 技術方針

## 6.1 フレームワーク
- Next.js を採用する

## 6.2 レンダリング方針
- 基本はフロントエンド主体の画面構成とする
- 再生操作やプレイリスト制御はクライアント側で完結させる

## 6.3 データ管理方針
- 楽曲データ、プレイリスト、再生キューはローカル管理とする
- 外部 API との通信は行わない
- 永続化が必要な場合はブラウザストレージ利用を将来拡張として検討する

## 6.4 設計方針
- Presentation と State を分離する
- ビジネスルールは usecase に集約する
- UI コンポーネントは表示責務のみに寄せる
- state 層は再生状態と UI 状態を保持する
- usecase 層は再生順制御やシャッフル判定などの振る舞いを管理する

---

## 7. 画面要件

## 7.1 画面構成概要
画面は大きく以下の 3 領域で構成する。

1. メイン再生エリア
2. 再生コントロールエリア
3. 右側の次曲リストエリア

Spotify や Apple Music のように、現在再生中の楽曲を中心に見せる構成とする。  
画面の主役は「現在再生中の曲」とし、右側には次に再生される予定の曲を補助情報として配置する。

## 7.2 レイアウト方針
- 左から中央にかけてメイン再生画面を広めに取る
- 右側に次曲リスト用のサイドパネルを配置する
- リストはトグル操作で表示 / 非表示を切り替えられる
- リスト非表示時はメイン再生画面を広く表示する

## 7.3 メイン再生画面
表示内容:
- ジャケット表示領域
- 曲名
- アーティスト名
- プレイリスト名
- 再生位置の視覚表示
- 主要操作ボタン

要件:
- 現在再生中の曲が明確に視認できること
- 曲情報は階層を持って表示すること
- Material Design の余白、角丸、階層感を持たせること

## 7.4 再生コントロール
表示内容:
- 再生 / 一時停止
- 次の曲へ送る
- シャッフル ON / OFF
- 次曲リスト表示切替

要件:
- アイコン主体で直感的に操作できること
- 操作状態が視覚的にわかること
- シャッフル有効時は状態が明確に見えること

## 7.5 次曲リスト
表示内容:
- 次に再生予定の楽曲一覧
- 各曲の簡易情報
- 現在の再生キュー順

要件:
- 右側に配置すること
- 表示 / 非表示が切り替えられること
- 現在再生中曲との差分が明確であること
- 次に来る曲ほど上位に表示すること

## 8. Material Design 準拠方針

### 8.1 デザイン原則
- 明確な階層構造
- 一貫した余白
- 状態変化のわかりやすさ
- 過剰装飾を避けたシンプルな UI
- インタラクション時の視覚フィードバック

### 8.2 コンポーネント方針
- ボタンは Material Design の押下感と状態変化を持つ
- パネルはカード的な整理された見た目にする
- 再生中状態、選択状態、無効状態を区別する
- 右側リストはサイドシートに近い見せ方にする

### 8.3 配色方針
- ダーク基調または中間色基調を推奨
- 再生中の曲や有効な操作はアクセントカラーで強調
- 背景、カード、強調色のコントラストを十分に確保する

### 8.4 タイポグラフィ
- 曲名は強調
- アーティスト名、プレイリスト名は補助情報として表示
- リスト項目は読みやすさ優先で情報量を絞る

## 9. 機能要件

## 9.1 楽曲再生
- ユーザーは楽曲を再生できる
- 現在再生中の楽曲情報が表示される
- 一時停止状態を持てる

## 9.2 次の曲へ送る
- ユーザーは現在の曲をスキップし、次の曲へ進められる
- 次曲決定はキュー順またはシャッフルルールに従う

## 9.3 シャッフル
- ユーザーはシャッフルの ON / OFF を切り替えられる
- シャッフル ON 時は再生順がランダム化される
- 現在再生中の曲は即時再抽選しない
- 次曲リストはシャッフル後の再生予定順に合わせて表示する

## 9.4 次曲リスト表示切替
- ユーザーは次曲リストを表示 / 非表示できる
- 非表示時も再生ロジックには影響しない
- 表示状態は UI state として管理する

## 9.5 再生キュー表示
- 現在の再生順を UI に反映する
- 現在再生中以降の曲のみ右リストに表示する
- キュー内容はシャッフル ON / OFF に応じて変動する

---

## 10. 非機能要件

## 10.1 操作性
- 主要操作は 1 クリックまたは 1 タップで完結する
- 複雑なメニュー階層は避ける

## 10.2 拡張性
- プレイリスト追加
- 前の曲へ戻る
- ループ再生
- シークバー操作
- 複数プレイリスト管理
- ストレージ保存

上記を後から追加しやすい構造にする

## 10.3 保守性
- UI とロジックの責務が明確であること
- usecase に業務ルールを集約し、変更箇所を限定すること
- 状態定義を一元的に追いやすい構成にすること

## 10.4 一貫性
- 画面全体で同じ操作体系を採用する
- Material Design の考え方に沿った見た目と挙動を保つ

## 11. アーキテクチャ設計

## 11.1 レイヤ構成
本アプリは以下のレイヤに分ける。

1. Presentation 層
2. State 層
3. Usecase 層
4. Domain 層
5. Infrastructure 相当のローカル実装層

### 11.1.1 Presentation 層
責務:
- 画面表示
- ユーザー操作受付
- 状態の見た目への反映

特徴:
- 表示中心
- ビジネスロジックを持たない
- 再生順の決定などは行わない

### 11.1.2 State 層
責務:
- 現在の再生状態保持
- UI 状態保持
- プレイリスト状態保持
- usecase 実行結果の反映

保持対象例:
- 現在再生中の曲
- 再生中 / 一時停止
- シャッフル ON / OFF
- キュー
- リスト表示 / 非表示

### 11.1.3 Usecase 層
責務:
- 次曲決定
- シャッフル順生成
- キュー再構築
- 再生開始条件整理
- UI 操作に対する業務ルールの実行

方針:
- ビジネスロジックを usecase に集約する
- Presentation から直接ロジック判断を行わない
- State 層は usecase の結果を反映する

### 11.1.4 Domain 層
責務:
- 楽曲
- プレイリスト
- 再生キュー
- 再生モード

などの概念整理

### 11.1.5 ローカル実装層
責務:
- ローカルデータの取得
- 初期プレイリスト供給
- 必要に応じたブラウザストレージ連携

## 13. 状態設計

## 13.1 管理対象 state
- 現在再生中の曲 ID
- 再生状態
- 現在のプレイリスト
- 再生キュー
- シャッフル状態
- 右側リスト表示状態
- 現在曲のインデックス
- 次曲一覧

## 13.2 state 分類
### 13.2.1 Player State
- 再生中か
- 現在曲は何か
- 現在キューは何か
- 次曲候補は何か

### 13.2.2 UI State
- 右リストが開いているか
- ボタン状態の見せ方
- 選択中 / 有効中の視覚状態

## 13.3 state 管理原則
- UI 表示都合の state と再生制御 state を分ける
- usecase が state を更新する際のルールを統一する
- Presentation 層で複雑な状態遷移を行わない

---

## 14. Usecase 設計

## 14.1 Usecase 一覧
- 再生開始 usecase
- 一時停止 usecase
- 次曲送り usecase
- シャッフル切替 usecase
- キュー構築 usecase
- 次曲一覧取得 usecase
- リスト表示切替 usecase

## 14.2 各 usecase の責務

### 14.2.1 再生開始 usecase
- 現在曲を再生状態へ移行する
- 必要ならキュー初期化を行う

### 14.2.2 一時停止 usecase
- 再生状態を停止系へ切り替える
- 曲情報やキューは保持する

### 14.2.3 次曲送り usecase
- 現在キューから次曲を決定する
- 現在曲を更新する
- 再生終了後の次候補を更新する

### 14.2.4 シャッフル切替 usecase
- シャッフル状態を切り替える
- キューを再構成する
- 現在曲を保持したまま次曲順を再計算する

### 14.2.5 キュー構築 usecase
- プレイリストから再生順を構築する
- シャッフル ON / OFF 条件を反映する
- 現在曲基準で次曲リストを生成する

### 14.2.6 次曲一覧取得 usecase
- UI 用に右側表示リストを返す
- 現在曲を除外した表示順を返す

### 14.2.7 リスト表示切替 usecase
- UI state のサイドパネル開閉を切り替える

## 20. 処理フロー

## 20.1 初期表示フロー
1. プレイリストをロードする
2. state を初期化する
3. 初期キューを構築する
4. 画面へ反映する

## 20.2 次曲送りフロー
1. ユーザーが次曲送り操作を行う
2. 次曲送り usecase を実行する
3. usecase が次に再生すべき曲を決定する
4. state を更新する
5. UI が現在曲表示と右側リストを再描画する

## 20.3 シャッフル切替フロー
1. ユーザーがシャッフルを切り替える
2. シャッフル切替 usecase を実行する
3. 現在曲を維持したままキューを再構築する
4. state を更新する
5. 右側リスト表示内容を更新する

## 20.4 リスト表示切替フロー
1. ユーザーがリスト表示切替を行う
2. UI state を更新する
3. サイドパネルの表示 / 非表示を切り替える

---

## 21. エラーハンドリング方針

- プレイリストが空の場合は再生不可状態を表示する
- 次曲が存在しない場合は末尾到達として扱う
- ジャケットがない場合はプレースホルダーを表示する
- 再生対象が無効な場合は UI 上で案内を出す

---

## 22. アクセシビリティ方針

- アイコンボタンには意味がわかるラベルを付与する
- キーボード操作に配慮する
- 再生中状態や有効状態は色だけに依存しない
- リスト表示 / 非表示状態を支援技術でも認識しやすくする

---

## 23. テスト観点

## 23.1 UI テスト観点
- 再生画面が正しく表示される
- 右側リストの表示 / 非表示が切り替わる
- 再生中曲が正しく表示される
- シャッフル状態が視覚的に反映される

## 23.2 usecase テスト観点
- 次曲送りで正しい曲へ進む
- シャッフル ON でキューが再構成される
- 現在曲を維持しつつ次曲順だけ再計算される
- 次曲一覧取得が現在曲を除いて返す

## 23.3 state テスト観点
- 再生状態更新が正しく保持される
- UI state と Player state が混在しない
- リスト表示状態が正しく切り替わる

## 27. 結論

本アプリは Next.js を用いたフロントエンド完結型の作曲・プレイリスト再生アプリとして設計する。  
UI は Spotify や Apple Music のような再生中心の画面をベースにしつつ、Material Design に準拠したシンプルな体験を提供する。

また、以下を設計上の中核方針とする。

- API は作成しない
- UI と state を分離する
- ビジネスロジックは usecase に集約する
- 右側に次曲リストを配置し、表示 / 非表示を切り替え可能にする
- 次曲送りとシャッフルを標準機能として備える

以上により、見た目のわかりやすさと、将来的な拡張のしやすさを両立する。
